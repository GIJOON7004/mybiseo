# 재해 복구 및 데이터 보존 정책 (DR-POLICY)

**문서 버전:** 1.0  
**최종 수정:** 2026-05-11  
**적용 대상:** 마이비서(MY비서) SEO SaaS 플랫폼  
**문서 소유자:** 기술 운영팀

---

## 1. 목적 및 적용 범위

본 문서는 마이비서 플랫폼의 데이터 보존 기간, 백업 전략, 재해 복구 목표(RTO/RPO)를 정의합니다. 마이비서는 의료기관(병원, 치과, 한의원 등)을 주요 고객으로 하는 SEO 진단 SaaS이며, 진단 보고서에 포함되는 데이터가 의료기관의 마케팅 의사결정에 활용되므로 적절한 보존 정책이 필요합니다.

> **중요 구분:** 마이비서는 전자의무기록(EMR) 시스템이 아닙니다. 환자 진료 데이터를 직접 취급하지 않으며, 의료법 시행규칙 제15조의 진료기록부 보존 의무(10년)가 직접 적용되지 않습니다. 다만 고객사가 의료기관이므로, 해당 기관의 컴플라이언스 요구에 부합하는 수준의 데이터 관리를 지향합니다.

## 2. 데이터 분류 및 보존 기간

| 데이터 유형 | 설명 | 보존 기간 | 근거 |
|------------|------|----------|------|
| SEO 진단 보고서 | HTML/PDF 리포트, 점수 이력 | 3년 | 사업 판단 (마케팅 트렌드 주기) |
| 키워드 분석 데이터 | 네이버 검색량, 경쟁도 스냅샷 | 2년 | API 데이터 재취득 가능 |
| 사용자 계정 정보 | OAuth 프로필, 이메일, 이름 | 계정 삭제 후 30일 | 개인정보보호법 제21조 |
| 결제/구독 이력 | 플랜 변경, 결제 기록 | 5년 | 전자상거래법 제6조 (거래기록 보존) |
| 시스템 로그 | API 호출 로그, 에러 로그 | 90일 | 운영 판단 |
| AI 생성 콘텐츠 | 블로그 초안, SEO 제안 | 1년 | 사업 판단 |

### 2.1 의료법 참조 (간접 적용)

의료법 시행규칙 제15조에 따른 진료기록 보존 기간은 다음과 같습니다 [1]:

- 진료기록부: 10년
- 환자 명부: 5년
- 처방전: 2년
- 수술기록: 10년
- 검사소견기록: 5년

마이비서가 직접 보관하는 데이터에는 해당하지 않으나, 고객사가 마이비서 보고서를 내부 의사결정 근거로 활용할 경우를 대비하여 SEO 진단 보고서의 보존 기간을 3년으로 설정합니다.

## 3. 인프라 및 SLA

### 3.1 데이터베이스: TiDB Cloud Serverless

마이비서는 TiDB Cloud Serverless를 주 데이터베이스로 사용합니다.

| 항목 | 내용 |
|------|------|
| 제공사 | PingCAP (TiDB Cloud) |
| SLA | 99.99% 월간 가용성 [2] |
| 데이터 복제 | Multi-AZ Raft 합의 알고리즘 (3중 복제) |
| 자동 백업 | 일일 자동 백업 (PingCAP 관리) |
| 백업 보존 | 7일 (Serverless 기본) |
| 암호화 | TLS 전송 암호화 + AES-256 저장 암호화 |
| 리전 | AWS ap-northeast-1 (도쿄) |

### 3.2 파일 스토리지: S3 호환 오브젝트 스토리지

PDF 보고서, 이미지 등 비정형 데이터는 S3 호환 스토리지에 저장됩니다.

| 항목 | 내용 |
|------|------|
| 내구성 | 99.999999999% (11 nines) |
| 가용성 | 99.99% |
| 버전 관리 | 비활성 (비용 최적화) |

## 4. 재해 복구 목표

| 지표 | 목표 | 설명 |
|------|------|------|
| **RPO** (Recovery Point Objective) | ≤ 24시간 | 최대 데이터 손실 허용 범위. TiDB 일일 백업 기준 |
| **RTO** (Recovery Time Objective) | ≤ 4시간 | 서비스 복구까지 최대 허용 시간 |
| **MTTR** (Mean Time To Repair) | ≤ 2시간 | 평균 복구 시간 목표 |

### 4.1 복구 시나리오별 대응

| 시나리오 | 영향도 | RPO | RTO | 대응 절차 |
|----------|--------|-----|-----|----------|
| 단일 노드 장애 | 낮음 | 0 | 자동 (< 30초) | TiDB Raft 자동 failover |
| AZ 장애 | 중간 | 0 | < 5분 | Multi-AZ 자동 전환 |
| 리전 장애 | 높음 | ≤ 24시간 | ≤ 4시간 | 백업에서 신규 리전 복원 |
| 데이터 오염 (논리적) | 높음 | ≤ 24시간 | ≤ 4시간 | PITR 또는 일일 백업 복원 |
| 랜섬웨어/보안 침해 | 심각 | ≤ 24시간 | ≤ 8시간 | 격리 → 백업 복원 → 포렌식 |

## 5. 백업 전략

### 5.1 현재 구현

- **TiDB 자동 백업:** PingCAP 관리형, 일 1회, 7일 보존
- **S3 오브젝트:** 삭제 시 소프트 삭제 (DB 참조 제거), 물리 삭제 없음
- **코드/설정:** GitHub Private Repository + Manus 체크포인트 시스템

### 5.2 향후 강화 계획 (Phase 2)

- 주간 논리 백업 (mysqldump 호환) → 별도 S3 버킷 30일 보존
- Cross-region 복제 (서울 리전 추가 시)
- 자동 복구 테스트 (분기 1회)

## 6. 데이터 삭제 정책

| 트리거 | 대상 | 절차 | 완료 기한 |
|--------|------|------|----------|
| 사용자 계정 삭제 요청 | 계정 + 연관 데이터 | 소프트 삭제 → 30일 유예 → 물리 삭제 | 요청 후 30일 |
| 보존 기간 만료 | 해당 유형 데이터 | 자동 배치 삭제 (heartbeat job) | 만료 익일 |
| 고객사 계약 해지 | 해당 기관 전체 데이터 | 소프트 삭제 → 90일 유예 → 물리 삭제 | 해지 후 90일 |

## 7. 개인정보 보호 준수

마이비서는 개인정보보호법 및 정보통신망법을 준수합니다.

- **수집 최소화:** OAuth 인증에 필요한 최소 정보만 수집 (이름, 이메일)
- **목적 외 이용 금지:** SEO 분석 목적으로만 데이터 활용
- **파기:** 보존 기간 경과 시 지체 없이 파기
- **안전성 확보:** TLS 1.3 전송 암호화, AES-256 저장 암호화
